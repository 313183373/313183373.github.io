ReactRouter例子ModalGallery引发的问题

Created at 2018-10-18 Updated at 2018-10-18 Tag React / ReactRouter

在看React Router 官方的EXAMPLES：Modal Gallery的时候，一开始没有看得太懂，这个例子展示了可以在相同的URL下展示不同的组件，它的原理呢，就是判断你是如何到达某个URL的页面的，比如说：有一个URL是/img/1，我们可以直接在地址栏输入这个URL进入页面，也可以点击一个Link组件进入（这与那个例子的方式不一样，不过原理相同）。

本来其实这两种方式没有区别，关键在于如果通过Link组件进入，我们可以给Link组件的to属性赋值为一个object，为什么是object呢？因为object可以传递一个state到location，我们在这里做一个标记，然后就可以判断是从哪里进入页面的，如果我们可以在location 中得到我们传递的state，那我们就知道是从Link进来的，然后我们就渲染不同的界面。

原理就是这样😂，本来很简单的，只不过那个例子中有一些干扰的东西，让我反应了半天🤪，现在我们来看看那个例子吧！

先给大家演示一下这个小程序的样子：

大家注意看url的变化，在主界面点击Image的时候，会是全屏幕显示那个背景色的图片。但是如果进入先到Gallery中，然后点击Gallery中的那些Image，就会变成在一个Modal中显示，而且原来的Gallery界面也还在。是不是很神奇。

接下来我们先来自己看看如何实现这个效果（我们不用写CSS，简单的写一个ID数字就可以，很轻松）。

我们先写一个普通版本的，也就是同一个的URL对应相同的页面。

现在这个程序很简单，我们在Home和Gallery组件中都加入了Link组件，都是去/img/:id的，注意我们在Galler中的Link是带有state的，但是我们现在还没有使用。下面我们将利用这个state来实现不同的渲染。我们可以在那个<Route path='/img/:id'>中判断当前的location的state是什么来判断我们是否从Gallery中的Link进入，从而实现我们想要的效果。

但是目前App组件是无法获取到location的，因为它不是Route渲染的，我们可以利用withRouter高阶组件来让它可以获取到当前的location。

主要看那个Route的写法<Route path='/img/:id' component={(location.state && location.state.fromGallery) && Modal || Img}/>

如果是从Gallery进来的，我们就渲染Modal组件，现在我们来看看效果！

可以看到我们实现了在同一个URL下渲染不同的界面的功能，但是感觉和官网的例子有点不一样，官网在显示Modal组件的时候，Gallery其实是可以看到的，这是怎么做到的呢？这就是那个例子中迷惑我的地方😂，那个例子中有一些和我们刚才讲的东西无关的代码，我不知道有什么用，后来我知道了，其实人家注释也写了，我来给大家慢慢道来，要实现那个效果，得用到Switch组件的，我们都知道Switch组件可以根据当前的location来匹配其中的一个Route来渲染，但是我还不知道其实Switch可以手动给location让它去匹配，这就是关键，我们可以把原来那三个Route用一个Switch包起来，同样当location有state的时候，我们可以给Switch一个location，让它渲染Gallery，同时呢，我们在Switch外，再写一个Route，同样匹配/img/:id，渲染Modal组件，这样我们就实现了渲染Modal的时候同时渲染Gallery了，来看看吧。

好的，目前我们已经实现了和官方例子一样的效果了，但是如果大家看了官方的代码的话，还是可以看到和我们程序的区别的，区别在于，我们这个程序是当从Gallery点进来的时候，把Switch的location给写死成gallery了，这样是不好的，如果我们有多个入口都可以进入那种显示Modal的状态，那么我们这么写就不行了，我们得动态的存储进入Modal状态的前一个状态的location，不太好理解？我们先来看官方的代码

其中重要的部分是App组件初始化了一个previousLocation属性，值就是当前的路径，其实这个地方有点问题的，因为我们初次渲染整个组件的时候，previousLocation讲道理应该是没有的，这里初始化为当前的Location，待会会给大家看会有什么问题😎

然后最关键的一点的是componentWillUpdate生命周期方法，这里判断只要当前不是Modal状态和下一个操作不是POP（这里我还不太理解，我暂时认为只有返回操作会是POP）就会把当前的location赋值给previousLocation，这就记录了上一个location 了，但是这样和我之前分析的意思不太一样，我们并不是每个previousLocation都需要记录，只需要记录下一个location是Modal状态就可以了，不过效果是确实实现了，我们添加了一个GalleryTwo组件，就是另一个Modal状态的入口，来看看效果

注意看刷新之后，原本是Modal状态下的/img/2，变成了正常状态的/img/2，这也没有问题，但是我们再返回一下，就出现问题了，一个奇怪的界面出现了，正常状态的/img/2和Modal状态的/img/1同时出现，这是为什么呢，因为刷新之后，其实location的state是保存着的，开始的/img/2是正常的，是因为render方法判断了previousLocation和当前location是相等时，就不是Modal状态，渲染了正常的/img/2（这就是我之前说的用当前location初始化previousLocation会出现的问题），一旦我们执行返回操作，previousLocation和location就不等了，而此时location的state也存在，就判断为了Modal状态，然后呢previousLocation是/img/2，也就出现了2和1同时出现的状态。
